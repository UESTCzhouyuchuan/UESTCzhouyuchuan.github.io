              # 如何自定义指令

              使用 ts 自定义指令注意 需要为 Cypress 命名空间添加指令，调用时语法提示才不会报错（虽然正常运行，但是看着不舒服），eslint 默认允许自定义指令，disable 一下就好。（和指令代码放一起就行）

              /* eslint-disable @typescript-eslint/no-namespace */
declare namespace Cypress {
  interface Chainable<Subject> {
    uploadFile(value: string): Chainable<Subject>;
  }
}

              # 第一步，放到正确的位置

              把自定义指令写到或者引入到 cypress/support/command.js 文件中。command.js 回被引入到 cypress/support/index.js 中，在测试开始前会自动执行。 import '../utils/upload';

              # 第二步，写指令。

              # 语法

              Cypress.Commands.add(name, callbackFn);
Cypress.Commands.add(name, options, callbackFn);
Cypress.Commands.overwrite(name, callbackFn);

              # name

              name 就是起的指令的名称。

              # option

              仅支持 add 时使用，不支持指令重新。 选项：

              • prevSubject, boolean,string or array
                • false,忽略前者对象，可以直接在 cy 上调用。默认值。
                • true, 接收前一个对象到回掉函数中，回掉函数的一个参数位置要留给它。而且因为要接受一个 subject，所以不能直接在 cy 上调用。
                • optional，结合上面两个，就可以接受对象，也可以不接受。
                • element，需要前者是节点元素
                • document，前者是 document
                • window，前者是 window 对象 参考文档：https://docs.cypress.io/zh-cn/api/cypress-api/custom-commands.html#Syntax

              # 如何判断是否属于一个完整的 cy 语句。

              遵循一下原则

              • 能放在一个 cy 中就尽量放在一个 cy 中。
              • 当使用 get/contain 时，链式调用返回的 DOM 就会缩小范围，当下一步获取的 DOM 非其子节点时要新起一个 cy。 关于如何选择 ele 节点
              1. 节点的 class 属性和 css 属性很有可能会发生变化，遵循一下三个原则：
              • 不要通过 CSS 属性如，id，class，tag 获得元素对象
              • 不要通过可变文本获得元素对象
              • 通过为 ele 增加 data-*属性获得 elem 对象
              <div data-cy="wrap-box">box</div>
              1
              1. 对于一个 textContent，判断是否推荐使用 contains。
              • 当 textContent 元素发生改编后，你是否想要得到一个找取失败的结果。
              • 如果是，推荐使用 cy.contains()
              • 如果不是，推荐使用 data-cy 属性。 关于返回值。
              • 不要尝试缓存函数的返回值，cy 的每一步操作都是异步的（类 promise），放进异步队列执行，不要以编写同步代码的习惯编写 cypress。而且 cypress 每一条语句（无论是 cy 开头还是 it）都会在上一条语句执行之后才会执行。
              • 如果使用返回值，使用类似 promise 的 then 的 closure 法，通过链式调用获得结果。如下
              cy.get('.name').then($name => {
  // $name is the object that the previous
  // command yielded us
});

              • 通常在 cypress 中你几乎不会用到const let var来声明变量，因为 cypress 是通过闭包很容易就可以获得你想要的数据。 Tip: You’re trying to store and compare values such as text, classes, attributes。
                • 有一个例外，当下一个状态依赖于上一个状态而且需要多次操作时，必须要记录下上一个状态。例如,一个点一下自增 1 的按钮。
              // cypress test code
cy.get('#num').then($span => {
  // capture what num is right now
  const num1 = parseFloat($span.text());

  cy.get('button')
    .click()
    .then(() => {
      // now capture it again
      const num2 = parseFloat($span.text());

      // make sure it's what we expected
      expect(num2).to.eq(num1 + 1);
    });
});

              • 当你在 cypress 的生命周期函数中如 before/befoeEach 记录一个初始数据，然后在接下的操作中使用。需要使用变量缓存数据。
              • 对于上一种需要缓存变量的情况，两个在不同的 it 域中，可以使用共享上下文的方式解决，主要设计 as,get(也可以使用 this 但是不推荐)。注意点，as 为异步操作，若以同步的方式去使用，无法达到想要的结果。对于路由别名，可以使用 cy.wait 去同步获得。例如：
              it('is not using aliases correctly', function() {

  cy.fixture('users.json').as('users');
  // this.users is not defined
  // because the 'as' command has only
  // been enqueued - it has not run yet
  const user = this.users[0];
  // 改为
  cy.fixture('users.json').then(users => {
    const user = users[0];
  });
});

              当有很多测试用例时

              • 尝试把每个测试用例单独测试（使用 only），所有用例都能单独通过而不依靠其他测试结束的状态，这才是最佳实践。
              • 如果单独测试无法通过，尝试把依赖的部分放到生命周期 hook 中或者合并测试。

              # 把每个断言作为一个小测试

              不要把断言写成像单元测试一样分工明确，意思是尽量把断言放在一个 it 里面，这样的话 cypress 就会少执行一些 lifecircle event，reset 一个 it 测试的时间要慢于注入一个断言，这样提高执行效率。

              # 使用 before 和 beforeEach 生命周期函数。

              before 会在所以 it 执行前执行一次，而 before 会在每个 it 执行前执行。

              • 例如，你想要测试咱们的 AST-stage 环境，需要在 before hook 里面先执行 setCookie 操作，登陆你的角色。
              context('test', () => {
  // 生命周期函数，只执行一次
  before(() => {
    cy.setCookie('people-session-7', 'XN0YXJ0-xizfztn2fazlydd5ea3l6dxmf5463rop-WVuZA');
  });
  it('a test case', () => {
    // cy.visit('https://people-staging.bytedance.net/atsx/talent/');
    cy.visit('/');
  });
});

              • 使用 before/beoferEach 去 clean state，而不是 after/afterEach，因为一下几个原因
                • 在 cyprss 执行过程中会因为存储每一个状态，这样做 not necessary。
                • cypress 执行过程中可能会出现 run fail，这样 after/afterEach 就无法执行，无法达到目的。

              # 不必要的的 wait

              对于 cypress 来说一切皆异步，异步的本质就是状态，所以 cypress 通过判断动作执行状态，很容易决定是否应该进行下一步，而不需要通过 wait 这种笨拙而且不合理的方式等待状态变化。cy.request，cy.visit，cy.get，都没必要添加 wait 等待时间。正确的做法如下。

              cy.server();
cy.route('GET', /users/, [{ name: 'Maggy' }, { name: 'Joan' }]).as('getUsers');
cy.get('#fetch').click();
cy.wait('@getUsers'); // <--- wait explicitly for this route to finish
cy.get('table tr').should('have.length', 2);
